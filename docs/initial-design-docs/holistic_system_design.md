# Holistic System Design (AI Vibe-Coding Optimized)

> **Context:** This document outlines the absolute highest-level software architecture for the TradeBias Greenfield project. Because this codebase will be heavily generated by AI (Vibe-Coded via AGY), the architecture *must* prioritize strict modularity, decoupled boundaries, and zero-lifetime spaghetti. 
> 
> **The Prime Directive:** If any specific component (e.g., the Polars Matrix engine, or the Genetic Algorithm) fails to work, we must be able to delete its folder and swap it out without rewriting the rest of the application.

---

## 1. The Multi-Crate Workspace (Forced Isolation)
To guarantee modularity and prevent "Spaghetti Code" (which causes AI context windows to hallucinate), the project will not be a single monolithic crate. It will be a **Cargo Workspace** composed of strictly isolated micro-crates.

```text
tradebias/
├── Cargo.toml                (Workspace definition)
├── crates/
│   ├── tb_core/              (Base types, Data models, Interfaces/Traits)
│   ├── tb_data/              (Data fetching, Polars preprocessing, Dataset stitching)
│   ├── tb_foundry/           (Phase 1: Genetic Algorithm, AST mutation, Matrix Math)
│   ├── tb_simulator/         (Phase 2: Execution, Walk-Forward Optimization, Risk)
│   └── tb_ui/                (egui, egui-charts, user interaction)
└── src/                      (The thin binary executable that links the crates)
```

**Why this is perfect for AI:** 
If you ask the AI to "Fix the crossover logic in the GA," the AI only needs to read `tb_foundry`. It doesn't need to know how the UI renders or how the Data is fetched. Forced boundaries equal perfect AI generation.

---

## 2. Trait-Driven "Plug & Play" Boundaries
To ensure we can swap out pipeline mechanics, crates will communicate *only* through Rust `trait` boundaries defined in `tb_core`. 

### The Pipeline Interfaces
If we decide later that Polars isn't fast enough and we want to rewrite Phase 1 using raw CUDA or `faer`, `tb_simulator` (Phase 2) shouldn't have to change a single line of code.

```rust
// In tb_core::interfaces

/// Phase 1 produces a stream of Elite Strategies.
pub trait AlphaGenerator {
    fn generate_elites(&self, data: &MasterDataset, config: &Phase1Config) -> Receiver<EliteStrategy>;
}

/// Phase 2 consumes an Elite Strategy and returns a Tear Sheet.
pub trait ExecutionSimulator {
    fn simulate(&self, strategy: EliteStrategy, data: &MasterDataset, config: &Phase2Config) -> TearSheet;
}
```
**The Rule:** Components never pass concrete struct implementations to each other. They pass Traits and Channels. This means you can have a `PolarsAlphaGenerator` and a `MockAlphaGenerator` side-by-side, hot-swapping them in the `main.rs` file.

## 3. Data Flow & Concurrency (The Asynchronous Pipeline)
AI code generators struggle immensely with complex Rust lifetimes (`<'a>`) and shared mutable state (`Arc<Mutex<T>>`). Therefore, the holistic architecture strictly uses **Data-Oriented Ownership**.

### The Message-Passing Architecture (Zero Shared State)
Instead of Phase 1 and Phase 2 trying to read and write to the same memory block, we use **Crossbeam Channels**.

1. **The Data Stage:** `tb_data` loads the CSV, builds the Polars DataFrame, computes all indicators, and yields a single, read-only `Arc<MasterDataset>`. 
2. **The Generation Stage:** `tb_foundry` takes a clone of the `Arc` (cheap pointer clone), runs the GA, and spits owned `EliteStrategy` structs into a `crossbeam_channel::Sender`.
3. **The Validation Stage:** `tb_simulator` threads listen on a `crossbeam_channel::Receiver`. They take *ownership* of the `EliteStrategy`, run the stateful loop, and push the final `TearSheet` into the UI channel.

**The Rule:** Data flows strictly in one direction. There are no circular dependencies. Memory is passed by value (ownership transfer) or via read-only `Arc`. 

---

## 4. Abstract Syntax Tree (AST) Modularity
The core definition of a strategy (The Sketch) must be incredibly easy for the AI to parse, mutate, and execute.

* **Recursive Enums (No Dynamic Dispatch):** We use a recursive `Expr` enum (`enum Expr { CrossAbove { lhs: Box<Expr>, rhs: Box<Expr> }, Sma { ... } }`) rather than `Box<dyn Indicator>`. 
* **Why?** This perfectly matches the `json_ast_schema.md`. Recursive enums are memory-safe, trivially serializable to JSON/YAML, and allow infinite composability without needing a complex central indicator registry. AI models natively understand tree structures.

---

## 5. Persistence & Session Management
A full Phase 1 and Phase 2 run can take minutes. Results must not be lost when the app closes.
* **Tear Sheet Serialization:** `tb_simulator` outputs `TearSheet` objects. These must implement `serde::Serialize` and be saved to a local SQLite database (or Parquet file) along with their originating `SessionConfig`.
* **The Strategy Library:** The UI will feature a "Library" tab that queries the local SQLite DB to load past runs, allowing side-by-side comparison of `TearSheets` generated on different days.

---

## 6. UI Decoupling (The "Headless" Engine)
The UI (`tb_ui`) must be completely decoupled from the engine. 
* The engine must be able to run in a Headless (CLI-only) mode.
* The UI simply sends a `StartSimulation(Config)` message to the engine via a channel, and listens for `Update` messages coming back.
* **Fallback Plan:** If `egui` turns out to be too difficult to format nicely, this architecture guarantees we could easily swap `tb_ui` out for a web-based React frontend (using a Rust WebSocket server) without touching a single line of the trading logic.

---

## 7. Summary for AI Code Generation (Vibe Coding Rules)
Whenever interacting with an AI (like AGY) to build this project, enforce these rules:
1. **Never use lifetimes (`'a`) unless absolutely necessary.** Use `Arc` for shared read-only data, or pass by value.
2. **Depend on `tb_core`.** All data structures and traits live in `tb_core`. Other crates only depend on `tb_core`, never on each other.
3. **Keep functions pure.** Phase 1 and Phase 2 functions should take inputs and return outputs. They should not mutate hidden global state. 
4. **Use Channels for flow.** The pipeline is just a series of functions connected by crossbeam channels.
