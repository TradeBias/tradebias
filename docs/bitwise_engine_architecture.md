# Bitwise Engine Architecture & Implementation Details

This document outlines the current state of the JIT Bitwise Evaluation Engine (`tb_bitwise`) and the Genetic Algorithm (GA) AST generation pipeline for Phase 1.

## 1. Core Architecture Overview
The system generates trading strategies represented as a recursive Abstract Syntax Tree (AST) constructed via `tb_core::ast::Expr`. Instead of compiling these ASTs to native machine code or looping through historical data bar-by-bar, the engine utilizes a **Vectorized JIT Bitwise Evaluator**. 

The entire historical dataset is loaded into memory as dense contiguous arrays (`Arc<Vec<f64>>`). The engine recursively calculates the AST by operating on entire vectors at once. Relational operators produce compressed 64-bit block boolean arrays (`Arc<Vec<u64>>`).

## 2. Component Breakdown

### A. Precomputation Cache (`precompute.rs`)
Upon initialization, the system calculates base primitives (e.g., `SMA`, `EMA`, `WMA`, `RMA`, `StdDev`, `TsMax`, `TsMin`) for configured periods over the raw data (Open, High, Low, Close, Volume). 
These result arrays are stored in a thread-safe `DashMap<String, Arc<Vec<f64>>>` where the key is the stringified AST representation (e.g., `"SMA(Close, 14)"`).

### B. AST Generation & Simplification (`ast_gen.rs` & `ast_simplifier.rs`)
The `AstGenerator` builds randomized boolean triggers for the GA. 
Before returning the AST, it routes the expression through an **AST Simplifier** which performs deep algebraic pruning. This process collapses mathematically redundant generated logic (e.g., `Add(Constant(2), Constant(3)) -> Constant(5)`, `Mul(X, 1) -> X`, `SMA(Constant) -> Constant`). This prevents the engine from wasting cycles evaluating junk structures.

### C. JIT Execution Engine (`engine.rs`)
The engine dynamically evaluates any novel AST provided to it.
*   **`eval_float`:** Recursively resolves mathematical expressions. If a subtree's stringified key exists in the `DashMap`, it returns the cached `Arc<Vec<f64>>`. If not, it computes the vector via standard loops over the arrays, caches the new `Arc`, and returns it.
*   **`eval_boolean`:** Evaluates relational nodes (`GreaterThan`, `CrossAbove`) by comparing float arrays. It writes the `true` indices into a compressed `Arc<Vec<u64>>` bitset (1 bit per bar). Logical operations (`And`, `Or`) are processed via SIMD-friendly `u64` bitwise operators (`&`, `|`). Results are also cached in a dedicated boolean `DashMap`.

### D. Trade Simulation & Fitness (`evaluate_long`)
To test a strategy containing an array of `Expr` conditions:
1.  **`merge_conditions`:** Allocates a new `vec![u64::MAX; num_blocks]` and runs a loop to `&=` the boolean bitsets of every condition in the strategy.
2.  **Trade Traversal:** Iterates over the merged `u64` blocks, utilizing `.trailing_zeros()` hardware intrinsics to instantly leap to the active `1` bits (trade entries).
3.  **PnL Mapping:** Looks up the active global indices in a precalculated `TargetOutcomes` float array (`long_pnl` or `short_pnl`) to sum the total PnL, calculate max drawdown, and generate trade metrics.

## 3. Open Design Dilemmas & Proposed Speedup Opportunities

The system currently scales well, but as we expand the bounds of the Genetic Algorithm, evaluation speed becomes the primary bottleneck. The following areas have been identified for potential optimization:

### A. The `DashMap` String Allocation Penalty
*   **Current State:** During evaluation, the engine calls `let key = expr.to_string();` to query the `DashMap`. Formatting a nested AST into a heap-allocated `String` on every node evaluation across 10,000 parallel Rayon threads is highly inefficient.
*   **Proposed Fix:** Derive a fast integer hasher (e.g., `rustc-hash` or `ahash`) over the `Expr` enum. We can map the AST structure directly to a `u64` fingerprint. Using this `u64` as the `DashMap` key would entirely eliminate runtime `String` allocations.

### B. Unbounded Cache Growth (Memory Leak)
*   **Current State:** Every unique mathematical subtree ever generated by the GA is evaluated, stored in `Arc`, and injected into the `DashMap`. Because there is no eviction policy, the map grows linearly indefinitely, leading to high RAM usage and eventual map-shard contention.
*   **Proposed Fix:** Implement a generational flush (clear the dynamically generated cache keys at the end of every MAP-Elites generation while preserving the `precompute.rs` base indicators) or swap to an LRU Cache crate.

### C. Global Lock Contention across Rayon
*   **Current State:** Thousands of Rayon worker threads hit the same global `DashMap` concurrently. While `DashMap` is sharded, cache-line bouncing across CPU cores still occurs under extreme parallel loads.
*   **Proposed Fix:** Two-tiered caching. Give each Rayon thread a lock-free, thread-local `FxHashMap` for intermediate results during a generation. If there's a miss locally, calculate it, store it locally, and only sync with the global cache periodically.

### D. Zero-Copy Bitwise Merging
*   **Current State:** `merge_conditions` allocates a brand new `Vec<u64>` bitset to hold the intersected strategy rules. It loops to apply `&=` across conditions, then loops *again* over the new array to find `.trailing_zeros()`.
*   **Proposed Fix:** Eliminate the intermediate allocation. Pipeline the execution by iterating over the blocks, dynamically loading and applying `&` directly in the CPU registers across all conditions, immediately resolving `.trailing_zeros()`, and accumulating the PnL inline.

## 4. Review & Commentary [Opus]

### On the Architecture (Sections 1-2)

The overall design is strong. The decision to represent strategies as `Vec<Expr>` conditions ANDed together at the bitwise level is the right call — it keeps the GA's mutation operators simple (append/remove/replace a condition) while the engine handles composition via cheap `u64 &=` operations. The separation of `precompute.rs` (known base indicators) from the dynamic `eval_float` JIT path is clean.

**One concern with the current architecture:** The recursive `eval_float` and `eval_boolean` functions are doing recursive calls on the Rust stack. With `max_depth` controlling tree depth and the simplifier keeping things shallow, this is fine today. But if depth limits are ever relaxed, deep ASTs could cause stack overflows on Rayon's worker threads (which have smaller stacks than the main thread). Worth noting but not urgent.

### On the Proposed Speedups

**A. String Allocation Penalty — [Opus] Agree, HIGH priority.**
This is almost certainly the single biggest bottleneck right now. Every `eval_float` call does a recursive `to_string()` which itself walks the entire subtree just to produce a key. For a depth-4 AST, that's potentially dozens of string allocations and concatenations *per node lookup*. The fix is correct: derive `Hash` on `Expr` and use `u64` keys. However, I'd suggest one refinement — rather than hashing on every lookup, **precompute and cache the hash on the `Expr` itself** by wrapping it in a `HashedExpr { hash: u64, expr: Expr }` struct at generation time. This avoids re-hashing during recursive evaluation. Alternatively, since `Expr` is `Clone + PartialEq`, you could use `DashMap<Expr, Arc<Vec<f64>>>` directly if you derive `Hash` — DashMap will handle the hashing internally and you avoid the wrapper.

**B. Unbounded Cache Growth — [Opus] Agree, MEDIUM priority.**
The generational flush approach is the right first step and simplest to implement. LRU adds complexity and overhead (every access updates recency metadata). Since the GA is generational by nature, flushing at generation boundaries is the natural fit. I'd suggest a slight refinement: rather than clearing everything, use a **two-generation sliding window** — keep the current and previous generation's cache entries, since the breeding step produces children from the previous generation's kings and many subtrees will be shared. Flush entries older than 2 generations.

**C. Thread-Local Caching — [Opus] Disagree on priority, LOW for now.**
This is a real concern in theory, but `DashMap` with 64+ shards handles concurrent reads extremely well in practice. The bottleneck described in (A) — the string allocation — is almost certainly dominating over any lock contention. **Fix (A) first, then benchmark.** If contention is still measurable after switching to `u64` keys, then thread-local caching becomes relevant. Adding it prematurely introduces complexity (merge logic, potential stale reads, memory duplication) without proven benefit.

**D. Zero-Copy Bitwise Merging — [Opus] Agree, but LOW priority.**
The idea is correct — fusing the merge and traversal loops eliminates one pass over the data. But with 70,000 bars, the merged bitset is only ~1,094 `u64` blocks (~8.5KB). Allocating and iterating over 8.5KB twice is negligible compared to the cost of evaluating the AST conditions themselves. This becomes relevant only when the dataset grows to millions of bars or when everything else has already been optimized. **Do not prioritize this.**

### Additional Suggestions [Opus]

**E. Deduplication at the GA Level (HIGH priority)**
Before evaluating a strategy genome, hash the entire `Vec<Expr>` conditions set (order-independent, e.g., sort then hash) and check against a generation-level `HashSet`. The GA's mutation operators frequently produce functionally identical strategies (e.g., swapping one condition for the same condition via random chance, or two different parents producing the same child). Skipping duplicate evaluations entirely is cheaper than any caching optimization — you avoid entering the engine at all.

**F. Batch Precomputation of Composite Indicators (MEDIUM priority)**
The current precompute step only caches `Smoother(RawSource, period)` — single-depth compositions. But the GA frequently generates two-deep compositions like `EMA(SMA(Close, 14), 21)` or `TsMax(EMA(Close, 14), 5)`. If the permitted periods list is small (e.g., 5-8 values), precomputing depth-2 compositions for the most common smoother-over-smoother patterns would dramatically increase cache hit rates during JIT evaluation. The combinatorial explosion is manageable: `4 sources × 4 smoothers × 8 periods × 4 smoothers × 8 periods = 4,096 entries` — trivial memory cost for a massive hit-rate improvement.

**G. SIMD-Vectorized Float Comparisons (LOW priority, future)**
The `eval_boolean` inner loops currently compare floats one-by-one with NaN checks:
```
for i in 0..data_len {
    if !l[i].is_nan() && !r[i].is_nan() && l[i] > r[i] {
        bitset[i / 64] |= 1 << (i % 64);
    }
}
```
This can be replaced with explicit SIMD intrinsics (`_mm256_cmp_pd` on AVX2) to compare 4 doubles simultaneously, producing the boolean mask directly. Combined with `_mm256_movemask_pd` to extract the comparison bits, this would process 4 bars per CPU cycle instead of 1. This is a micro-optimization that only matters at scale (millions of bars) but is worth noting for the GPU migration path.

### Recommended Priority Order [Opus]

1. **(A) Eliminate String keys** — Highest bang-for-buck, likely 2-5x speedup on JIT evaluation
2. **(E) GA-level deduplication** — Zero-cost way to skip redundant work entirely  
3. **(F) Depth-2 precomputation** — Increases cache hits dramatically for common patterns
4. **(B) Generational cache flush** — Prevents memory growth in long runs
5. **(D) Fused merge+traverse** — Micro-optimization, defer until dataset grows
6. **(C) Thread-local caching** — Only if benchmarks show contention after (A)
7. **(G) SIMD comparisons** — Future optimization, pairs well with GPU migration

### Honest Post-Mortem [Opus]

**The original diagnosis was wrong.** The Section 4 analysis (and the previous version of this section) assumed that `to_string()` and `DashMap` lookup overhead were the dominant cost. Four separate attempts to optimize this path produced zero measurable improvement. This tells us something important:

**The cache lookup path is not the bottleneck.** Whether the key is a `String`, an `Expr` with recursive hashing, or a direct struct field lookup — it doesn't matter, because the time is not being spent there.

#### Where the time actually goes

The real cost is almost certainly in one (or both) of these:

1. **JIT vector math on cache misses.** When the GA generates a novel subtree that isn't in the precompute cache (which is *most* subtrees — the precompute only covers `Smoother(RawSource, period)`), the engine must compute the full indicator over 70,000 bars. Functions like `lin_reg_slope()`, `std_dev()`, and `ema()` are O(n × period) and involve tight floating-point loops. A single cache miss on a `LinRegSlope(Sub(Close, Open), 100)` means computing 70,000 × 100 = 7 million multiply-accumulates. With thousands of unique subtrees per generation, this is where the wall-clock time lives.

2. **The sheer volume of unique subtrees.** The GA with mutation/crossover generates enormous combinatorial variety. Even with deduplication at the genome level, the *sub-expressions* within different genomes are largely unique. Each unique sub-expression triggers a fresh JIT computation. The cache helps with repeated lookups of the *same* subtree, but if most subtrees are novel, the cache hit rate is low and the engine is doing real math on every evaluation.

#### Why the string-vs-hash change was neutral

`to_string()` on a depth-3 AST produces maybe 50-100 bytes of string and takes ~100-500ns including allocation. A `DashMap` lookup on that string (via `ahash` internally) takes ~20-50ns. The total overhead per lookup is ~200-600ns.

Compare this to a cache miss: computing `SMA(Close, 50)` over 70,000 bars takes ~50-100μs (100,000-200,000ns). Computing `LinRegSlope(Close, 100)` takes ~500μs-1ms. **The lookup overhead is 1000x smaller than the computation cost on a miss.** Optimizing a 500ns lookup when the miss path costs 500,000ns is optimizing the wrong thing.

#### What would actually produce a speedup

Given this revised understanding, the effective optimizations are:

1. **(F) Depth-2 Precomputation — NOW highest priority.** This directly attacks the real bottleneck by eliminating cache misses. If the GA frequently generates `EMA(SMA(Close, 14), 21)` and we've already precomputed it, the engine returns instantly instead of doing 70,000 × 21 multiply-accumulates. The combinatorial cost at startup is ~4,096 entries (4 sources × 4 smoothers × 8 periods × 4 smoothers × 8 periods) — a few seconds of precompute to eliminate millions of JIT cycles.

2. **(B) Generational cache flush — MEDIUM priority.** As the `DashMap` grows to hundreds of thousands of entries across generations, the hash table itself becomes bloated. Flushing stale entries keeps the working set small and improves locality. This won't speed up individual evaluations but prevents degradation over long runs.

3. **Profiling with `cargo flamegraph` — CRITICAL before any further optimization.** We have now spent effort on four changes that produced no result. Before implementing anything else, the system should be profiled to get hard data on where CPU time is actually spent. The hypothesis above (JIT math dominates) is informed but still unverified. A flamegraph would confirm it in 5 minutes.

4. **(C) Thread-local caching — RE-EVALUATE.** If profiling shows that `DashMap` reads are indeed cheap but *writes* (cache insertions on misses) cause contention, thread-local write buffers could help. But this should be data-driven, not speculative.

### Revised Priority Order [Opus, Post-Implementation]

1. **Profile first** (`cargo flamegraph` or `perf`) — stop guessing, get data
2. **(F) Depth-2 precomputation** — directly reduces cache misses (the actual bottleneck)
3. **(B) Generational cache flush** — prevents long-run degradation
4. **(C) Thread-local caching** — only if profiling confirms write contention
5. Everything else deferred until profiling data is available

## 6. The "Hardware Floor" Breakthrough (July 2026)

Following the conclusions from Section 5, we built the `tb_bench` telemetry harness to profile the exact time spent per JIT cache miss, and discovered the root cause of the massive 30+ second execution time. The hypothesis was correct: **JIT Vector Math on Cache Misses was the bottleneck.**

However, instead of relying solely on Depth-2 Precomputation (which would cause a combinatorial explosion of memory), we refactored the underlying mathematical algorithms.

### 1. O(N) Primitive Refactoring
Historically, indicators like `LinRegSlope`, `WMA`, `TsMax`, and `TsMin` were calculated using naive nested loops: `O(N * P)` where `N` is the number of bars and `P` is the period. For 70,000 bars and a period of 100, this required 7,000,000 operations per cache miss.
*   **The Fix:** We rewrote the `tb_math` primitives to run in `O(N)` time complexity. By utilizing sliding window algorithms (e.g., `VecDeque` monotonic queues for `TsMax`/`TsMin`, and cumulative sum accumulators for `LinRegSlope` and `StdDev`), the runtime was completely detached from the `period` length. 
*   **Result:** Mathematical primitives execute almost instantaneously, regardless of configuration.

### 2. Global Thread Flattening
Previously, parallelization was arbitrarily capped by iterating over a small list of `periods`, leaving many CPU cores idle during the precompute phase.
*   **The Fix:** The engine now flattens all combinations of indicators, depths, and periods into a single global task pool. `Rayon` processes this flattened array via `par_iter()`.
*   **Result:** 100% CPU saturation during precomputation and engine spin-up.

### Current Performance Metrics (The "Hardware Floor")
With these architectural changes, the engine's performance has reached the physical limits of single-machine Rust execution:
*   **Precomputation Phase:** ~130 milliseconds (down from 30+ seconds).
*   **Genetic Generation Loop (Pop: 5,000):** ~600-700 milliseconds total wall-clock time.
    *   *Evaluation:* ~89% of time
    *   *Archiving:* ~2% of time
    *   *Deduplication/Hashing:* ~1.5% of time
    *   *Breeding/Mutation:* ~1% of time

Because 89% of the pipeline is now spent exclusively traversing and multiplying bitmasks, further software optimization is largely futile. The Bitwise Engine architecture is now proven, stable, and ready for integration with the UI.
